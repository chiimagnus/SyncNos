## Summary

This commit introduces a cancelable sync task registry (`SyncRunningTaskStore`), makes the global concurrency limiter cancel-aware, adds de-duped “ensure database/properties” in `NotionSyncEngine`, and threads `source` through sync notifications to avoid cross-source collisions.

## Findings (Problems / Risks)

### 1) `ConcurrencyLimiter` can leak a permit on cancellation (high)

File: `SyncNos/Services/Core/ConcurrencyLimiter.swift`

In `acquire()`, there is a `try Task.checkCancellation()` *after* the waiter is resumed. If a waiter is resumed (meaning it effectively “received” a permit) and then the cancellation check throws *before* `withPermit` installs its `defer { release() }`, the permit may not be released.

Impact:
- Global concurrency can permanently shrink after a few such races, causing “sync tasks stuck forever” symptoms.

### 2) `ConcurrencyLimiter` cancellation handler has a race window (medium)

File: `SyncNos/Services/Core/ConcurrencyLimiter.swift`

`onCancel` schedules `cancelWaiter(id:)` asynchronously. If cancellation happens before the waiter is appended, `cancelWaiter` might not find it. This can leave a canceled waiter in the queue (hang).

Impact:
- Tasks can remain stuck in “waiting” despite user cancel.

### 3) Cancel window between “enqueue in SyncQueueStore” and “register in SyncRunningTaskStore” (medium)

Files:
- `SyncNos/ViewModels/*/*ViewModel.swift`
- `SyncNos/Services/SyncScheduling/*AutoSyncProvider.swift`

Tasks are enqueued (UI shows `.queued`) before the corresponding `Task` is registered into `SyncRunningTaskStore`. If the user cancels immediately during this window, the cancel request may not reach the underlying `Task`, and the work can still execute later.

Impact:
- User sees a task canceled, but it may still run and hit Notion/network.

### 4) `Cancelling...` is present but lacks translations (low)

File: `SyncNos/ViewModels/Sync/SyncQueueViewModel.swift`

`NSLocalizedString("Cancelling...", ...)` is used for the cancel UX hint. The key exists in `Resource/Localizable.xcstrings`, but it only has metadata/comment and no actual `localizations` entries in this commit.

Impact:
- i18n coverage gap; likely falls back to the raw key for most locales.

### 5) Minor diagnostics gaps (low)

File: `SyncNos/Services/SyncScheduling/SyncQueueStore.swift`

The “no task matched” warning doesn’t include the `source` value. With source filtering introduced, logs are harder to interpret without it.

## Suggested Fixes

1. Fix permit leak: remove the post-wait cancellation check from `acquire()`; instead perform `Task.checkCancellation()` inside `withPermit` after installing `defer { release() }`.
2. Fix cancellation race: track “cancelled waiter ids” (and/or “recently resumed ids”) so cancellation is honored even if it arrives before append or after resume.
3. Close cancel window: in each per-item sync `Task`, check `SyncQueueStore.isTaskActive(source:rawId:)` before waiting for a permit / doing work; if not active, exit early.
4. Add `Cancelling...` to `Resource/Localizable.xcstrings` (at least English + zh-Hans).
5. Improve warning logs by including `source`.

## Fixes Applied After Review (working tree)

The following fixes were applied locally after writing the findings above:

- Fixed `ConcurrencyLimiter` cancellation edge-cases (permit leak + cancellation race hardening).
- Added “task still active?” guard (`SyncQueueStore.isTaskActive`) at the start of per-item sync `Task` to close the cancel window.
- Added `source` into the “no task matched” warning log.
- Added `en` + `zh-Hans` translations for `Cancelling...` in `Resource/Localizable.xcstrings`.
